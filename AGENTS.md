# AGENTS.md

## 목적

이 저장소는 반복 지시 없이도 작업이 이어지도록 고정된 실행 워크플로우를 사용한다.

## 기준 문서 우선순위

작업 기준은 아래 순서를 따른다.

1. `TODO.md`
2. `NEXT_SESSION_PLAN.md`
3. `SPEC.md`
4. `README.md`

문서 간 충돌이 있으면 더 위 문서를 우선하고, 작업 중 하위 문서를 함께 정리한다.

## 기본 워크플로우

사용자가 `진행`, `계속`, `이어서`처럼 계속 작업을 요청하면 기본적으로 아래 순서대로 수행한다.

1. 계획 수립
   - 목표, 범위, 완료 기준, 검증 방법을 정한다.
   - `TODO.md`의 `Roadmap Loop Harness`에서 `active` 상태인 최상위 항목을 기본 대상으로 삼는다.
2. 계획 검토
   - 리스크, 회귀 가능성, 의존성 누락, 가정 불명확성을 먼저 점검한다.
   - 범위가 너무 크면 한 단계로 쪼갠다.
3. 작업 수행
   - 승인된 단계 범위 내에서 바로 구현한다.
   - 중단 조건이 아니면 추가 확인을 기다리지 않는다.
4. 검증
   - 가장 작은 정적 검증과 런타임 검증부터 수행한다.
   - 필요하면 더 넓은 회귀 테스트까지 확장한다.
5. 문서 반영
   - 동작, 상태, 우선순위가 바뀌면 `README.md`, `SPEC.md`, `TODO.md`, `NEXT_SESSION_PLAN.md`를 갱신한다.
6. 커밋
   - 검증이 끝난 단위로 집중된 커밋을 만든다.
7. 다음 항목 자동 진행
   - 중단 조건이 없으면 현재 `active`를 닫고, 첫 번째 `pending`을 다음 `active`로 승격해 바로 진행한다.

## 루프 상태 모델

- `active`: 현재 세션의 유일한 최상위 실행 항목
- `pending`: 다음 자동 진행 후보
- `blocked`: blocker와 재개 조건이 기록된 대기 항목
- `done`: 구현, 검증, 문서, 커밋까지 끝난 항목
- `archived`: 잠정 중단 또는 이력 보존 항목

추가 규칙:
- `active` 항목은 항상 정확히 1개만 유지한다.
- `blocked`로 이동할 때는 `TODO.md`에 blocker와 재개 조건을 함께 남긴다.
- `archived` 항목은 실행 대상으로 다시 올리지 않는다. 사용자가 직접 재개를 지시한 경우에만 예외로 본다.

## 역할 정책

모든 작업은 아래 역할을 순차적으로 수행하는 것으로 본다.

- 기획자: 범위, 완료 기준, 테스트 계획 정의
- 리뷰어: 결함, 회귀, 누락 가정 선점검
- 작업자: 코드 또는 문서 변경 수행
- 검수자: 검증 실행 및 결과 정리
- 유지보수자: 문서, 체크리스트, 커밋 상태 정리

이 역할들은 절차적 구분일 뿐이며, 사용자에게 각 역할별 추가 지시를 다시 요구하지 않는다.

## 자율 진행 정책

아래 조건에서는 사용자 확인 없이 계속 진행한다.

- 다음 작업이 `TODO.md` 또는 `NEXT_SESSION_PLAN.md`에 이미 정해져 있는 경우
- `scripts/roadmap_harness.py status` 결과로 현재 `active` 항목이 명확한 경우
- 변경이 로컬 저장소 범위 안에서 끝나는 경우
- 검증 방법이 기존 테스트나 스크립트로 명확한 경우
- 현재 단계 완료를 위해 필요한 작은 문서, 설정, 테스트 보완인 경우

## 중단 조건

아래 중 하나에 해당하면 멈추고 사용자에게 확인한다.

- 파괴적이거나 되돌리기 어려운 작업이 필요한 경우
- 제품 범위, 아키텍처, 사용자 동작이 크게 바뀌는 경우
- 우선순위 충돌이 있는데 문서 기준으로 해결되지 않는 경우
- 자격증명, 유료 서비스, 외부 시스템, 권한 상승이 필요한 경우
- 검증을 할 수 없고 결과를 자신 있게 확정할 수 없는 경우

## MVP 우선 정책

사용자가 별도 범위 변경을 지시하지 않는 한, MVP는 아래 기준을 따른다.

- 기본 경로: 웹 UI 기반 로컬 RAG
- 포함 범위: `intro -> app/admin`, 질의, 문서 조회, 업로드 요청, 관리자 승인
- 기본 제외: CLI 제품화, 데스크톱 래핑, GraphRAG 통합

`TODO.md` 우선순위가 바뀌거나 사용자가 직접 요청하지 않는 한, CLI, 데스크톱, GraphRAG로 범위를 넓히지 않는다.

## 검증 정책

단계를 닫기 전에 최소한 아래를 수행한다.

- 변경 파일 기준 문법 또는 정적 검증
- 변경 영역 기준 타깃 테스트
- 메인 사용자 경로에 영향이 있으면 가능한 범위의 전체 회귀 테스트
- 검증이 막히면 막힌 이유와 영향 범위를 명시

## 완료 정책

아래 조건이 모두 만족돼야 한 단계가 완료된 것으로 본다.

- 구현이 반영되었다.
- 관련 검증이 실행되었거나, 검증 blocker가 명확히 기록되었다.
- 상태 문서가 갱신되었다.
- 작업이 커밋되었다.

## 사람 인계 정책

사람과 에이전트가 같은 기준을 쓰도록 `WORKFLOW.md`를 이 파일과 맞춰 유지한다.

# Desktop Packaging Hardening Review (2026-03-17)

## 목적
- Electron 데스크톱 PoC를 패키징/배포 하드닝 단계로 바로 밀어도 되는지 재검토한다.
- MVP 포함 여부가 아니라 "지금 추가 투자할 가치가 있는가"를 판단한다.

## 현재 상태
- 현재 구현 경로는 `desktop/electron` 단일 PoC다.
- 검증 가능한 스크립트는 아래까지만 있다.
  - `npm run check`
  - `npm run preflight`
  - `npm run smoke`
- `package.json`에는 패키징 스크립트와 배포 도구(`electron-builder`, `electron-forge`)가 없다.
- 앱 런타임은 여전히 아래 외부 준비 상태를 전제한다.
  - Python 런타임
  - Python 의존성 설치
  - 로컬 LLM 런타임(Ollama 등)
  - 임베딩 모델 캐시 또는 로컬 경로

## 하드닝 관점 점검

### 가능한 것
1. preflight로 "왜 실행이 안 되는지"를 앱 시작 전에 드러낼 수 있다.
2. Electron이 직접 띄운 Python 서버 lifecycle을 최소 수준으로 관리할 수 있다.
3. 브라우저 기본 UI를 그대로 쓰므로 프론트 중복 개발은 피할 수 있다.

### 아직 비어 있는 것
1. 패키징 도구 부재
- 설치 파일 생성 경로가 아직 없다.

2. Python 배포 전략 미정
- embedded Python으로 번들할지, 별도 설치를 요구할지 정해지지 않았다.

3. LLM/임베딩 런타임 번들 전략 미정
- Ollama 모델과 embedding cache는 설치형 패키지 범위를 크게 키운다.

4. 플랫폼 운영 항목 미정
- 코드 서명, 업데이트, 로그 수집, 장애 복구 정책이 없다.

## 선택지 비교

### A. Electron + embedded Python
- 장점:
  - 사용자 설치 경험은 가장 단순해질 수 있다.
- 단점:
  - 앱 크기 증가
  - 플랫폼별 번들/서명 복잡도 증가
  - 모델/임베딩까지 포함하면 MVP 범위를 크게 넘는다

### B. Electron + 별도 Python/LLM 설치 요구
- 장점:
  - 지금 구조와 가장 가깝다.
- 단점:
  - "설치형 앱"이 아니라 사실상 런처에 가깝다.
  - 사용자 실패 지점이 많고 PoC 대비 가치가 작다.

### C. 브라우저 기본 경로 유지
- 장점:
  - 현재 MVP와 가장 일관된다.
  - 운영 이슈를 늘리지 않는다.
- 단점:
  - 데스크톱 실행 경험은 당분간 제공하지 못한다.

## 판단
- 결론: 패키징/배포 하드닝은 지금 단계에서 추가 진행하지 않고 보류한다.
- 이유:
  1. 가장 큰 미결정 사항은 UI가 아니라 Python/LLM 배포 전략이다.
  2. 그 결정 없이 패키징만 추가하면 "실행 파일은 있으나 실제 준비 항목은 그대로 많은" 상태가 된다.
  3. 현재 우선순위인 업로드 운영 흐름/GraphRAG 판단 대비 사용자 가치가 낮다.

## MVP 반영 판단
- MVP 기본 경로에는 넣지 않는다.
- 데스크톱 경로는 계속 실험용 PoC로만 유지한다.

## 재개 조건
1. `embedded Python` vs `별도 설치 요구` 중 하나를 먼저 결정한다.
2. Ollama/임베딩 준비 방식을 설치 가이드 또는 번들 전략으로 고정한다.
3. 그 다음에만 `electron-builder` 또는 동등 도구 도입을 검토한다.

# Version Boundary Reset (2026-04-04)

## 목적
- 현재까지의 버전 관리 서사와 실제 코드/평가/문서 구조 사이의 어긋남을 명시한다.
- `V1 본체`, `sample pack`, `archive`의 경계를 다시 고정한다.
- 이후 구현/문서/평가에서 같은 혼선을 반복하지 않도록 분류 기준을 남긴다.

## 현재 진단
- 현재 `trunk_rag`는 문서상으로는 `범용 local RAG product`를 지향해 왔다.
- 그러나 실제 코드와 평가 게이트는 유럽 과학사 샘플셋에 강하게 결합돼 있었다.
- 따라서 현재까지의 버전 관리에는 "제품 본체"와 "샘플/실험"의 경계가 느슨했던 문제가 있었다.

## 왜곡이 발생한 이유
### 1. 샘플 데이터셋이 제품 구조를 결정했다
- `eu/fr/ge/it/uk` 컬렉션이 코드 상수와 운영 문서에 직접 박혔다.
- 이 때문에 샘플 데이터셋이 제품의 기본 구조처럼 굳어졌다.

### 2. 샘플 질문셋이 운영 게이트가 됐다
- `ops-baseline`이 사실상 유럽사 3문항 answer-level 평가로 운용됐다.
- 그 결과 "샘플셋 통과"가 "제품 품질 보장"처럼 해석될 위험이 커졌다.

### 3. 샘플 질문 패턴이 로직까지 침투했다
- 질문 유형별 후처리(`역할/비교/상징`)가 특정 평가셋 보정으로 사용됐다.
- 이는 제품 본체의 일반 품질 개선이 아니라 샘플셋 최적화에 더 가깝다.

### 4. 아카이브와 본체 문서가 충분히 분리되지 않았다
- GraphRAG 이력, 유럽사 샘플, 운영 MVP 문서가 한 흐름 안에 섞였다.
- 보존은 됐지만, "현재 기준"과 "과거 실험"이 같은 레벨에서 읽히게 됐다.

## 재분류 원칙
- 제품 본체에 들어가는 것은 특정 도메인 없이 재사용 가능한 기능이어야 한다.
- 샘플 데이터셋은 제품 검증 예시일 뿐, 제품 구조를 강제하면 안 된다.
- 과거 실험/판단 이력은 필요하지만 기본 운영 문서와 분리돼야 한다.

## 재분류표
| 구분 | 포함 대상 | 현재 예시 | 처리 원칙 |
| --- | --- | --- | --- |
| `V1 본체` | 도메인 독립 기능, 운영 기본 경로, 공통 API/UX | `run_doc_rag.bat`, `/intro`, `/app`, `/query`, upload/admin, runtime profile, fingerprint, trace/meta | 유지 및 안정화 |
| `Sample Pack` | 샘플 데이터셋, 샘플 질문셋, 샘플 라우팅 규칙 | `eu/fr/ge/it/uk` 문서셋, 유럽사 answer fixtures, 역사/국가 예시 | 본체에서 분리, 선택형으로 격하 |
| `Archive` | 과거 실험, PoC, 도입 판단 이력 | GraphRAG 문서/리포트, 데스크톱 패키징 재검토 이력 | 보존하되 본체 기준 문서와 분리 |

## 즉시 반영해야 할 해석
### `V1 본체`
- 로컬/폐쇄망 RAG 런타임
- 웹 MVP 기본 경로
- 인덱싱/질의/업로드 요청/승인
- 운영 게이트와 복구 경로

### `Sample Pack`
- 유럽 과학사 Markdown 세트
- `eu/fr/ge/it/uk` 컬렉션 정책
- 유럽사 질문 기반 answer-level eval
- 국가명 키워드 라우팅

### `Archive`
- GraphRAG snapshot/sidecar 실험
- GraphRAG 질문셋과 Go/No-Go 판단
- 데스크톱 제품화 보류 판단 이력

## 앞으로의 문서 원칙
1. README/SPEC/ROADMAP의 기본 주어는 항상 제품 본체로 유지한다.
2. 샘플 데이터셋 설명은 별도 문서나 `sample pack` 섹션으로 내린다.
3. 아카이브 이력은 현재 우선순위 문맥에서 분리한다.

## 앞으로의 구현 원칙
1. 제품 본체는 `dataset-agnostic` 방향을 우선한다.
2. 샘플 데이터셋 전용 로직은 feature flag 또는 외부 manifest 뒤로 이동한다.
3. 샘플셋 성능 향상은 본체 품질 향상으로 간주하지 않는다.

## 결론
- 지금까지의 버전 관리가 완전히 무의미했던 것은 아니다.
- 다만 `제품`, `샘플`, `아카이브`의 경계가 느슨했고, 그 결과 코드와 문서의 버전 서사가 실제보다 더 일반적인 제품처럼 보이게 된 문제가 있었다.
- 2026-04-04 이후 기준에서는 이 세 층을 분리해 해석하는 것을 공식 기준으로 삼는다.
